# カスタム Web ブロッカー — 取扱説明書

これは、拡張機能の完全なリファレンス マニュアルです。最も簡単で一般的なワークフローから始まり、徐々にカスタム イベント ドリブンのブロック ルールやヘルパー API などの高度なトピックに移行します。

初めての方は、**クイック スタート** と **ブロック グループの概要**をお読みください。これらのセクションの下にあるものはすべて、やりたいことに応じてオプションです。

---

## 1. この拡張機能の機能

カスタム Web ブロッカーを使用すると、自分で定義したルールに従って Web サイトやオンラインの気を散らすものをブロックできます。次のことができます。

- ブラウザーのネイティブ ネットワーク ブロック (`ERR_BLOCKED_BY_CLIENT` を生成するのと同じ種類のブロック) を使用してサイトを即座にブロックします。
- サイトで 1 日あたり一定の分数を許可し、その制限を超えたらブロックします。
- ユーチューブ、ティックトック、フェイスブック、インスタグラム、ツイッチ、レディット 上の特定の種類のコンテンツをブロックします (サイト全体ではありません)。
- 単一のページのみをブロックするのではなく、サポートされているプラ​​ットフォームのフィードからブロックされたコンテンツを非表示にします。
- 曜日および `HHMM-HHMM` 時間枠ごとにルールがアクティブになるスケジュールを設定します。
- ルールを凍結して、簡単に変更できないようにします。厳密なフリーズでは、指定された時間ロックされ、元に戻すには 20 段階の確認手順が必要です。
- ルールを一時的にスヌーズしますが、十分な長さの理由を書いた後にのみ行います。
- 前方/後方タイマー、グループごとの永続ストレージ、プラットフォームごとの DOM インテント (ナビゲーション ボタンの非表示、述語によるフィード カードの非表示、サブセクションごとのタイマーの設定)、URL ユーティリティ、および構造化ログのヘルパーを使用して、スクリプト言語 で **イベント ドリブン** カスタム ルールを作成します。
- 50 を超える既製のテンプレート (タイマー、スケジュール、フィード非表示、フォーカス セッション、リダイレクト、ナッジ、永続性、DOM 調整、デバッグ ヘルパー) の組み込みライブラリから選択します。
- 20 以上の言語で拡張機能を使用します。

この拡張機能は、1 つのエディター ページ (ポップアップ)、1 つのバックグラウンド サービス ワーカー、カスタム ルール コードをホストする 1 つのオフスクリーン サンドボックス、およびすべてのページで実行される 1 つのコンテンツ スクリプトを備えた クローム Manifest V3 拡張機能です。カスタム ルールはオフスクリーン サンドボックスに存在します。これらは、「実行」クリックごとに 1 回ロードされ、ルールが無効になるか削除されるまで登録されたままになります。

## 2. UI ツアー

拡張機能のアイコンをクリックすると、エディターが完全な Web ページ (小さなポップアップではなく) として開きます。このページには次の領域があります。

- **トップバー**
  ・**取扱説明書**ボタン（本書）
  - **言語**ピッカー
  - **設定**ギア (**デバッグモード**を含む高度な切り替え)
- **左側のパネル — ブロック グループ**
  - ブロック グループのリスト。各カードには、グループ名、短い概要行、および有効化/無効化チェックボックスが表示されます。
  - **追加** ボタンは新しいグループを作成します。その横にあるドロップダウンでタイプを選択します。
  - **すべて削除** では、すべてのグループが削除され、グループが凍結されている場合は追加の確認が行われます。
  - カードの `::` ハンドルを上下にドラッグして、グループの順序を変更できます。
  - 垂直スプリッターをドラッグして、このパネルのサイズを変更できます。
- **右パネル — エディタ**
  - 現在選択されているグループを編集します: 名前、ブロック動作、ブロックリスト、タイプ固有のフィルター、スケジュール、フリーズ、スヌーズ。
  - すべての変更は、入力または操作を停止した後、数秒以内に自動的に保存されます。
  - **カスタム** グループの場合、エディターには **テンプレート** ブラウザ、**実行** ボタン、および **ログ** パネル (v1.1 の *アクティビティ ログ* から名前変更されました) も表示されます。
- **トースト** (フェードアウトする中央のポップアップ) - 「変更を保存しました」などのステータス メッセージを表示します。または入力ミス。
- **ページ内オーバーレイ** — タブにアクティブなタイマーまたはブロックがある間、タブの左上隅にオーバーレイが表示され、それに影響するすべての制約が `hh:mm:ss` (または `mm:ss`) 形式で表示されます。複数の制約は複数の行に積み重ねられます。デフォルトのブロックグループのカウントダウンとカスタムルールタイマーはこのオーバーレイを共有します。

## 3. クイックスタート1. 拡張機能アイコンをクリックします。エディターがページ全体として開きます。
2. [**ブロック グループ**] パネルで、ドロップダウンからグループ タイプを選択します。
   - `Default`、`ユーチューブ`、`ティックトック`、`フェイスブック`、`インスタグラム`、`ツイッチ`、`レディット`、または `Custom`。
3. [**追加**] をクリックします。新しいグループが表示され、エディターがそれを開きます。
4. 名前を付けます。
5. タイプ固有のフィールドに入力します (`Default` の場合、**ブロックされた Web サイト** リストを意味します)。
6. 左側のパネルにあるグループのチェックボックスがオンになっていることを確認します。
7. リストされているサイトのいずれかにアクセスします。ブロックはすぐに有効になります。

それが幸せな道全体です。このマニュアルの残りの部分は、これに加えられた単なるオプションです。

> カスタム グループで **実行** を押すと、新しいルールが **将来**のページ イベントに適用されます。すでに開いているタブでは、再ロードするまで前のルールが実行され続けます。実行が成功するたびに、ポップアップにその旨のリマインダーが表示されます。

## 4. ブロックグループの概要

この拡張機能内のすべては **ブロック グループ** として編成されています。ブロック グループは 1 つのルール セットです。

- 名前、タイプ、および有効/無効の状態があります。
- ブロック動作があります (即時、数分後、または固定カウントダウン)。
- オプションのスケジュール (日 + 時間枠) とオプションのフリーズ/スヌーズ制御があります。
- タイプに応じて、Web サイトのリスト、ユーチューブ クリエイター フィルター、サブレディット名、イベント駆動型 スクリプト言語 ルールなどの追加フィールドがあります。

グループはいくつでも持つことができます。複数のグループを同じページに適用できます。その場合、**最も厳格な** ルールが優先されます。

- 「すぐにブロック」は「しばらくしてからブロック」よりも優れています。
- 残り時間が少ないグループが、残り時間が多いグループに勝ちます。

したがって、さらにグループを追加すると、ページ ブロックがすぐに作成されるだけで、後で作成されることはありません。

**評価順序は下から上です。** 拡張機能がブロック グループを反復処理するとき、リストの一番下のグループから開始して上に向かって作業を進めます。リストの一番上のグループが最後に評価され、「最後の単語」を取得します。たとえば、一番下のグループが `helpers.getPlatformHelper().youtube().hideShortButton()` を呼び出し、一番上のグループが `showShortButton()` を呼び出した場合、ボタンは表示されたままになります。この順序を変更するには、カード上の `::` ハンドルをドラッグします。

## 5. グループの種類

### 5.1 `Default` — 通常の Web サイトをブロックする

特定のドメインをブロックする場合 (一般的な使用例)。

- **ブロックされた Web サイト**: 1 行に 1 つのサイト。 `facebook.com` と `https://www.facebook.com/somepage` は両方とも機能します。拡張機能はホスト名を抽出して正規化します。
- サイト ルールは、そのホスト名とそのすべてのサブドメインに適用されます。
- このグループ タイプは、`ERR_BLOCKED_BY_CLIENT` と同様に、クローム のネイティブ ネットワーク ブロッキングを使用します。つまり、ブロックされた URL へのナビゲーションは、ページが読み込まれる前に停止されます。

### 5.2 `ユーチューブ` — ユーチューブ および類似のビデオ サイトをブロックします

**フィルター** セクションをエディターに追加します。

- **コンテンツ タイプ**:
  - `Apply to all ユーチューブ pages` — すべての ユーチューブ ページが重要です。
  - `Apply to Shorts` — ショート ページのみがカウントされます。
  - `Apply to long videos` — `/watch`、`/live/`、`/embed/` などのみ。
  - `Apply to ユーチューブ posts` — コミュニティ投稿 (`/post/...`、チャンネル コミュニティ/投稿タブ)。
- **著者フィルター**:
  - `Do not filter by author` — 著者の身元は関係ありません。
  - `Apply to certain authors` — リストされた著者のみがこのグループをトリガーします。
  - `Apply to all except certain authors` — リストされている著者は免除されます。
- **著者**: 1 行に 1 人の著者。 `@handle`、完全な URL、`/channel/UC...`、`/c/...`、`/user/...` を受け入れます。
- **ユーチューブ フィードでブロックされたエントリを非表示にする**: このグループがアクティブにブロックしている間、ユーチューブ フィード内の一致するカードは非表示になります。ブロックが非アクティブになると、次の更新時に戻ります。

ショートと投稿のコンテンツ タイプの場合、作成者フィルターが設定されておらず、グループが現在ブロックしている場合、拡張機能は関連するナビゲーション エントリ（ショートのサイドバー エントリ、コミュニティ/投稿チャンネル タブ）や、「最新の ユーチューブ 投稿」などの一致するシェルフも非表示にします。

ページ フォームが検出できる場合、短いものと長いものの検出は、ティックトック、Vimeo、ツイッチ クリップ/VOD、Dailymotion などの他のビデオ サイトにも拡張されます。

### 5.3 `ティックトック` — ティックトック コンテンツをブロックする

プラットフォームビデオエディターと同じエディターカードですが、ティックトック 固有のラベルが付いています。- コンテンツ タイプ: ショート ビデオ、ビデオ、プロフィール ページ。
- 著者: ティックトック ハンドル (`@handle`) またはプロフィール URL。
- フィード非表示では、グループがアクティブな間、ティックトック ページ上の一致するカードが非表示になります。

### 5.4 `フェイスブック` — フェイスブック コンテンツをブロックする

- コンテンツ タイプ: リール、ビデオ、投稿。
- 著者: ページ名 (`page.name`)、プロフィール URL、または `profile.php?id=...` フォーム (数値 ID は `id:<number>` として保存されます)。
- フィードを非表示にすると、フェイスブック 上で一致するフィード カードが非表示になります。

### 5.5 `インスタグラム` — インスタグラム コンテンツをブロックする

- コンテンツ タイプ: リール、ビデオ、投稿。
- 著者: インスタグラム のハンドル名またはプロフィールの URL。
- `/reel/`、`/p/`、`/tv/`、`/explore/` などの予約されたパスは作成者として扱われません。
- フィードを非表示にすると、インスタグラム 上で一致するカードが非表示になります。

### 5.6 `ツイッチ` — ツイッチ コンテンツをブロックする

- コンテンツ タイプ: クリップ、ストリーム/VOD、チャンネル ページ。
- 著者: チャンネル名またはチャンネル URL。
- `/directory`、`/videos`、`/settings` などの予約されたパスはチャネル名として扱われません。
- フィードを非表示にすると、ツイッチ で一致するカードが非表示になります。

### 5.7 `レディット` — レディット または特定のサブレディットをブロックする

- **サブレディット**: 1 行に 1 つのサブレディット。空のリストは、グループが レディット 全体に適用されることを意味します。 `productivity` と `r/productivity` の両方が受け入れられます。

### 5.8 `Custom` — イベント駆動型 スクリプト言語 によるブロック

ページを開く、URL の変更、ページのハートビート、タイマーの終了、独自のカスタム イベントなどのイベントの **ハンドラーを登録**する スクリプト言語 関数を作成します。この関数は、[実行] クリックごとに 1 回実行されます。登録されたハンドラーは、再度「実行」を押すか、グループを無効にするか、削除するまで、ナビゲーション間でアクティブなままになります。

`Custom` グループには、ブロック動作、ブロックされたサイト、許可された分数、リセット間隔、スケジュール日、または時間枠が表示されません。 **ブロッキング ルール** エディターと標準のフリーズ/スヌーズ コントロールが維持されます。また、パラメーター化されたスターター ルールを含むプリセット ブラウザーを開く [**テンプレート**] ボタンもあります。プリセットを適用すると、確認後に現在のルールが置き換えられます。

完全なカスタム ルールのリファレンスとヘルパー API については、**セクション 11** を参照してください。

## 6. ブロック動作

ほとんどのグループ タイプでは、3 つのモードのいずれかを選択します。

### 6.1 すぐにブロックする

ルールは、グループがオンであり、スケジュールで許可されており、(プラットフォーム グループの場合) ページが一致する場合に常にアクティブになります。

`Default` グループの場合、これは クローム のネイティブ ブロックを使用します。プラットフォーム グループの場合は、ページはめ込みオーバーレイ/終了ロジックを使用します。

### 6.2 数分後にブロックする

これは使用予算です。

- **ブロック前の許容分数** (10 進数): 1 ピリオドあたりに許可する分数。例: `15`、`0.5`、`90`。
- **タイマー リセット間隔 (時間)** (10 進数): 予算をリセットする頻度。例: 毎日の場合は `24`、時間ごとの場合は `1`、15 分ごとの場合は `0.25`。

時間が残っている間は、ページは通常どおり動作し、タイマー オーバーレイが表示されます。予算がゼロになると、ページは残りの期間ブロックされ、オーバーレイに `0:00` が表示され、タブは終了しようとします。

拡張子はグループごと、期間ごとです。

- 各グループには独自の予算があります。
- グループに一致するページに費やした時間は、そのグループの予算にカウントされます。
- 同じグループ内の複数のタブが予算を共有します。タイマーは常に同期されています。別のタブに切り替えると強制的に更新されるため、現在の共有時間がすぐに表示されます。

複数の時間制限グループが同じページに適用される場合、最も厳しいグループが優先されます。

### 6.3 タイマー (カウントダウンしてからブロック)

このモードでは、カウントダウン タイマーが表示され、`0:00` に達するとブロックされます。

- **タイマー リセット間隔 (時間)** (10 進数): タイマーの長さとリセット頻度の両方。例: 毎日の場合は `24`、時間ごとの場合は `1`、15 分ごとの場合は `0.25`。

**一定時間後にブロック**とは異なり、このモードには個別の「ブロックまでの許容時間」フィールドは**ありません**。タイマーは単純にリセット間隔で開始し、一致するページが開いている間カウントダウンし、次のリセットまでブロックします。デフォルト グループのカウントダウンとカスタム グループ タイマー (**セクション 11.3.1** を参照) はどちらも **タブが表示されている間のみ**進みます。タブを切り替えたり、ウィンドウを最小化したり、画面をロックすると、カウントダウンが自動的に一時停止されます。

## 7. スケジュール

**スケジュール** カードでは、グループがアクティブになる時間を制限できます。

- **ブロックする日**: グループが適用される日を選択します。チェックされていない日は、その日にグループが非アクティブであることを意味します。
- **時間ウィンドウ**: 自由形式のリスト。`HHMM-HHMM` 形式で 1 行に 1 つのウィンドウ。例:

  ZXQコード0ZXQ

  グループはそれらのウィンドウ内でのみアクティブになります。空のリストは終日を意味します。

これは、`Custom` を除くすべてのグループ タイプに適用されます。 (カスタム ルールは、`ev.time.dayName` / `ev.time.hour` を使用して独自のスケジュールを実装できます。**セクション 11.4** を参照してください。)

## 8. 凍結（改ざん防止）

凍結すると、グループを衝動的に無効にすることが難しくなります。

**フリーズ** カードで次のことを選択します。

- **凍結** — グループを編集または削除することはできません。また、その有効化トグルをオフにすることもできません。何かを変更するには、凍結解除の儀式を実行する必要があります (以下を参照)。
- **完全凍結** — 凍結と同じですが、選択した時間数 (10 進数で最大 72) ロックされたままになります。そのタイマーが切れるまでは、凍結解除の儀式さえも利用できません。

凍結されたグループのロックを解除できる場合、[**凍結解除**] ボタンが表示されます。それをクリックすると、**20 ステップの儀式**が始まります。

- モーダルは自己規律のメッセージを示します。
- `Confirm` を 20 回クリックする必要があります。
- クリックの間には強制的に 5 秒間の待機が発生します。
- 途中でキャンセルした場合は、手順 1 からやり直す必要があります。
- 20 個のメッセージがローテーションされるので、実際にメッセージを読むことができます。

グループが「スヌーズなし」(次のセクションを参照) とマークされている場合も、フリーズ中にスヌーズすることはできません。

フリーズ ステータスは、厳密なフリーズまでの残り時間を含む、グループ カードのメタ行に表示されます。

## 9. スヌーズ (一時的に無効にする)

スヌーズは、グループを解凍せずに一時的に無効にします。遅延アクティベーション、スヌーズ後のクールダウン、確認ステップ、およびスヌーズ時間の累計をサポートします。

**スヌーズ** カード内:

- **このグループのスヌーズを許可** — オフの場合、このグループはまったくスヌーズできません (フリーズ中も含む)。
- **スヌーズ期間 (分)** — スヌーズの継続時間を 10 進数で表します。
- **アクティブ化遅延 (分)** — 10 進数の `>= 0`。スヌーズを確認した後、グループはこの遅延が経過するまでブロックし続けます。そうして初めてスヌーズが有効になります。
- **スヌーズ後のクールダウン (分)** — `0` から `5` までの 10 進数。スヌーズが終了すると、クールダウンが終了するまで、このグループに対して別のスヌーズを開始することはできません。
- **確認回数** — 整数 `>= 0`。これが `0` の場合、スヌーズはすぐにスケジュールされます。それ以外の場合、スヌーズを開始すると、まさにその数のステップを含む確認の儀式が開始されます。

スヌーズ確認の各ステップでは、次のクリックが許可されるまで強制的に **5 秒間の待機**が行われます。モーダルはこれを明示的に示し、ボタンにライブカウントダウンを表示します。

グループがフリーズされている場合、スヌーズ設定はフリーズ前に選択された値にロックされます。スヌーズが許可されている限り、スヌーズすることはできますが、保存された遅延/クールダウン/確認設定を使用する必要があります。

スヌーズ カードには、そのグループの **合計スヌーズ時間** も表示されます。この合計には、その期間中に他の理由でサイトにアクセス可能になった場合でも、完全なアクティブ スヌーズ期間がカウントされます。

スヌーズが終了すると、ルールはすぐに戻ります。グループがまだフリーズされていない場合、拡張機能はスヌーズの終了時に自動的にグループを再度フリーズします。

スヌーズを確認するステータス メッセージが表示されます。スヌーズが終了すると、グループは自動的に通常の状態に戻ります。

**スヌーズ終了** ボタンを使用して、スヌーズを早めに終了することもできます。

カスタム グループの場合、**スヌーズ開始**を押すと、`snoozePress` イベントがルールに送出されます (**セクション 11** のイベント表を参照)。そのため、カスタム ルールは、プレスを記録したり、正当性をログに記録したり、フォローアップ イベントを起動したりできます。このルールには **プログラムによるスヌーズ API** はありません。報道に反応することはできますが、キャンセルしたり延長したりすることはできません。

## 10. 一括アクション- **すべて削除** はすべてのグループを削除します。
  - 常に確認を求められます。
  - 少なくとも 1 つのグループが凍結されている場合は、凍結解除と同じ 20 ステップの儀式が必要です。
  - いずれかのグループが完全に凍結され、まだロックされている場合、**すべて削除** は無効になります。

## 11. カスタム グループ — イベント駆動型リファレンス (v1.1 以降)

v1.1 以降、カスタム ルールは **イベント駆動型**です。ルールは、戻り値がページをブロックするハートビートごとの関数ではなくなりました。代わりに、ルール本体は、特定のイベント (ページを開く、URL の変更、ページのハートビート、カスタム イベントなど) の **ハンドラーを登録**するスクリプトです。ハンドラーは、ページ ナビゲーションやタブの切り替えを行っても登録されたままとなり、長期間存続する**オフスクリーン サンドボックス**内に存在します。

ルールの本文は、**実行クリックごとに 1 回** (または、グループが有効でアクティブなソースがすでに存在する場合は 1 回) 実行されます。ハンドラーを再ロードするには、エディターで [**実行**] をクリックします。ポップアップには、すでに開いているページをリロードして新しいルールが適用されるようにするよう求めるリマインダーが表示されます。

### 11.1 ルールの署名

```js
(event, helpers) => {
  // Register handlers here. This function is called exactly once
  // per Run click (or when the group is enabled).
}
```

2 つの引数:

- `event` — このグループの **イベント レジストリ**。これを使用して、ハンドラーの登録、オーバーライド、リスト、カウント、または登録解除、および `post(...)` カスタム イベントを行います。
- `helpers` — ヘルパー バンドル (**11.3** を参照)。

この関数は値を返すことを期待していません**。ブロックするか許可するかの決定は、後でイベントが発生し、登録されたハンドラーの 1 つが `ev.preventDefault()` および/または `ev.setResult(...)` を呼び出したときに行われます。

### 11.2 ライフサイクル

- **実行** (エディターのグループごとのボタン): エンジンはまず、以前にこのグループでタグ付けされたすべてのハンドラーを消去し、オフスクリーン サンドボックスでルール本体を再実行します。ソース編集後に再登録するにはこれが唯一の方法です。
- **グループを無効にする**: このグループでタグ付けされたすべてのハンドラーがワイプされます。グループ ソースはストレージ内に保持されますが、イベントへの応答を停止します。
- **グループを再有効化**: エンジンはこのグループのアクティブなソースを自動的に再実行します。
- **グループの削除**: 無効化と同じです。そのグループでタグ付けされたすべてのハンドラーがワイプされます。
- **同じ `(eventType, id)` で再登録**: 以前の登録がサイレントに上書きされます。

オフスクリーン サンドボックスは、**すべて**のカスタム グループによって共有されます。そこには、さまざまなグループのハンドラーが共存しており、それぞれが所有するグループ ID で内部的にタグ付けされているため、「実行」、無効化、または削除は正しいグループのみに適用されます。

カスタム ルールが誤動作した場合 (同期無限ループ、暴走ログ スパムなど)、サンドボックスによって隔離されます。グループは自動的に無効になり、失敗が記録されるため、[ログ] パネルで確認できます。隔離されたルールを再度有効にするには、ソースを修正して [**実行**] をクリックします。エンジンによって中止理由がクリアされ、ルールがリロードされます。

### 11.2.1 イベント レジストリ (`event`)

一般的な方法:

- `event.register(type, id, handler, options?)` — 任意のイベント タイプのハンドラーを登録します。 `id` はあなた自身の選択です。 `options.priority` (デフォルト `0`) — 高いものが最初に実行されます。 `options.intervalMs` — `tickEvent` のみ。グローバルティックに対してこの特定のハンドラーを調整します。同じ `(type, id)` で再登録するとオーバーライドされます。
- `event.unregister(type, id)`、`event.unregisterAll(type)`。
- `event.post(type, data?, { scope })` — カスタム イベントを起動します。 `scope: "global"` はすべてのグループに到達します。デフォルトの `scope: "group"` は、**同じ** グループ内のハンドラーにのみ到達します。

イベントごとのタイプのシュガー (組み込みタイプごとに 1 セットのメソッド):

- `event.registerTickEvent(id, handler, opts)`、`event.getTickEvent(id)`、`event.getTickEvents()`、`event.countTickRegistered()`。
- `event.registerOpenWebEvent(id, handler, opts)`、`event.getOpenWebEvent(id)`、`event.getOpenWebEvents()`、`event.countOpenWebRegistered()`。
・`closeWebEvent`、`switchWebEvent`、`switchDomainEvent`、`webChangedEvent`、`pageHeartbeatEvent`、`timerEnded`、`snoozePress`も同形状です。

### 11.2.2 組み込みイベントタイプ

|タイプ |発火するとき | `ev.data` ペイロード |
|---|---|---|
| `tickEvent` |ブラウザ全体でグローバルに共有される 1 秒のティック。タブの可視性に関係なく起動されます。これは、タブがフォーカスされていない場合でも実行し続ける必要がある時計スタイルのロジックに使用します。 | `{ intervalMs: 1000 }` |
| `pageHeartbeatEvent` | **アクティブ**、**表示**タブからのハートビートは約 250 ミリ秒。 `getOrCreateTimer({ scope })` に組み込まれた自動ティックを含む、タブの可視性を認識するすべてのロジックを駆動します。バックグラウンドタブや画面がロックされている間は**起動しません**。 | `{ elapsedMs }` |
| `openWebEvent` |新しいタブが作成されるか、エンジンがまだそのタブを認識していない URL に新しいナビゲーションが表示されます。 「実行」をクリックした後、すでに開いているタブに対して再起動は**されません**。 | `{ previousUrl, isNewTab }` |
| `closeWebEvent` |タブが閉じられています。 | `{ reason, nextUrl }` |
| `switchWebEvent` |同じタブ内の URL **変更** - 戻る/進む、SPA ルートの変更、または以前とは異なる URL に移動するナビゲーション。通常のリロード (同じ URL) では**起動しません**。 | `{ previousUrl, previousHostname, sameDomain }` |
| `switchDomainEvent` | URL の変更はホスト名の境界を越えます (例: `youtube.com` → `wikipedia.org`)。 `switchWebEvent` と一緒に起動します。 | `{ previousUrl, previousHostname }` |
| `webChangedEvent` |ページは、開く、切り替える、SPA 履歴の更新、**または同じ URL を維持する通常のリロード**など、あらゆる方法で (再) ロードされます。これは信頼できる「ページが変更されたので、すべてを再評価する」フックです。 `openWebEvent` / `switchWebEvent` / `switchDomainEvent` と一緒に起動され、同じ URL のリロードで起動する唯一のものです。 | `{ previousUrl, previousHostname, sameDomain, isFirstLoad, isReload, transition }` ここで、`transition` は、`"tabCreated"`、`"commit"`、または `"history"` です。 |
| `timerEnded` |グループによって管理されているタイマーが `currentMs === 0` に達しました。所有グループにのみ配信されます。 | `{ timerId, displayName, direction, currentMs }` |
| `snoozePress` |ユーザーは、この **カスタム** グループのポップアップで **スヌーズ開始**を押しました。純粋な通知イベント — ハンドラーは任意のコード (ログ、リダイレクト、他のイベントの起動) を実行できますが、カスタム ルールには **プログラムによるスヌーズ API** はありません。ここで生成されたログは、アクティブなタブにトーストとして表示されます。プレスされたグループにのみ配信されます。 | `{ triggeredAt }` |

`ev.url` およびイベント データ内の URL は、イベントに対して **正規化**されます。クローム の新しいタブ ページ (Google の「Google 検索または URL を入力」画面をレンダリングします)、`about:blank`、および同等の新しいタブ スキームは、空の文字列 `""` として公開されます。したがって、`ev.url === ""` をスコープとするタイマーは、新しいタブ ページを表示している間のみ作動します。通常の `google.com` URL は変更されません。

### 11.2.3 イベントオブジェクト (`ev`)

すべてのハンドラーは `(ev, helpers) => void` として呼び出されます。 `ev` には以下が含まれます:

- `ev.type` — ディスパッチされたイベントのタイプ。
- `ev.groupId` — 受信グループの ID。
- `ev.tabId`、`ev.pageId`、`ev.url`、`ev.hostname` — イベントのコンテキスト。
- `ev.time` — ディスパッチ時の `{ now, month, dayOfMonth, dayName, hour, minute }` スナップショット。 `dayName` は `"Sunday"`..`"Saturday"` です。
- `ev.data` — イベント固有のペイロード (上の表を参照)。

方法:

- `ev.preventDefault()` — ディスパッチを「ブロック」としてマークします。後で優先度の高いハンドラーが `setResult(1)` を設定しない限り、ホスト コンテンツ スクリプトはページを終了します (または `setRedirectLink` に従います)。
- `ev.stopPropagation()` — このディスパッチを直ちに停止します。 **このイベントに対しては、どのグループにもこれ以上のハンドラーは呼び出されません**。
- `ev.setResult(value)` — ディスパッチ結果を設定します。 `value` は、`[-255, 255]` の **数値** (`-1` ブロック、`0` ニュートラル、`1` 許可。他の整数は独自のデバッグ ロジック用に保存されます)、または **文字列** (リダイレクト URL として解釈されます) の場合があります。すべてのハンドラーにわたる最後の `setResult` 呼び出しが優先されます。数値 `1` は、以前の `preventDefault` をオーバーライドします。
- `ev.setRedirectLink(url)` / `ev.getRedirectLink()` — ディスパッチがブロックされて終了したときにホストが移動する必要がある URL。これはカスタム ルールからリダイレクトする**唯一**の方法です。エディターは、カスタム グループの「ブロックされた場合のリダイレクト URL」フィールドを公開しなくなりました。
- `ev.post(type, data, { scope })` — ハンドラー内からフォローアップ イベントを起動します。

さらに、`ev` はプロキシです。それに設定したフィールド (例: `ev.foo = 42`) は `custom` マップに保存され、同じハンドラーまたは同じディスパッチ内の後続のハンドラーから読み取ることができます。### 11.3 `helpers` オブジェクト

すべてのハンドラー呼び出しは、受信グループとイベントの URL をスコープとする新しい `helpers` バンドルを取得します。定数フィールド:

- `helpers.now` — ディスパッチ時のエポックミリ秒。
- `helpers.currentUrl` — newtab/blank 正規化後のイベント URL。
- `helpers.groupId` — グループ ID を受信して​​います。

便利なショートカット (以下のヘルパーで使用されるのと同じアキュムレータ対応関数にルーティングされるため、出力は依然として [ログ] パネルに表示されます)。

- `helpers.log(...)`、`helpers.warn(...)`、`helpers.error(...)`。

アクセサメソッド:

- `helpers.getLogHelper()` — `log` / `warn` / `error`。出力はレート制限されており、暴走ルールがポップアップをフリーズするのを防ぐためにディスパッチごとに制限されています。
- `helpers.getDomainHelper()` (別名 `helpers.getDomainUtility()`) — URL 検査 (**11.3.5** を参照)。
- `helpers.getTimerHelper()` — グループスコープのタイマー (カウントダウン / カウントアップ)。ブラウザを再起動しても状態は維持されます。
- `helpers.getPersistenceHelper()` — グループをスコープとする JSON キー/値ストア。
- `helpers.getRedirectionHelper()` — `setRedirectLink(url)` / `getRedirectLink()` (および `set` / `get` エイリアス) に加えて、指定されたメッセージを表示する `chrome-extension://...` URL を返す `createMessageUrl(message)`。
- `helpers.getPlatformHelper()` — プラットフォームごとの DOM インテント (**11.3.6** を参照)。
- `helpers.getDOMHelper()` — 一般的な DOM インテント: `hide(sel)`、`show(sel)`、`addClass(sel, c)`、`removeClass(sel, c)`、`setText(sel, text)`、`click(sel)`、`injectCss(css, id?)`、`removeInjectedCss(id)`、`scrollTo(sel)`。操作はバッチ化され、ハンドラーが戻った後に適用されます。
- `helpers.getNavigationHelper()` — `back()`、`forward()`、`reload()`、`goTo(url)`、`closeTab()`。エフェクトは、イベントが発生したタブに適用されます。
- `helpers.getStorageHelper()` — `getPersistenceHelper` のスーパーセットと、クロス拡張ストレージ用の非同期 `requestAsyncGet(key)` / `requestAsyncSet(key, value)` フック (結果はフォローアップ カスタム イベントとして到着します)。
- `helpers.getTabHelper()` — イベントにバンドルされているスナップショットに対する `list()`、`getActiveTab()`、`getById(id)`、`countOpen()`。

すべてのヘルパー メソッドは安全です。不正なパラメーターは、スローする代わりに `null`、`false`、または空の値を返します。

#### 11.3.1 `getTimerHelper()`

グループごとのタイマー。各タイマーは、選択した文字列 `id` によって識別されます。 ID のスコープはグループに設定されているため、2 つのグループは両方とも、衝突することなく ID `"yt-shorts"` を使用できます。ブラウザを再起動しても状態は維持されます。

タイマーの永続状態は、正確には、`id`、`displayName`、`direction` (`"forward"` または `"backward"`)、`isPaused`、および `currentMs` です。保存された「初期期間」はありません。`isExpired` は単なる `currentMs === 0` です。転送タイマーは永久に経過し、自動的に期限切れになることはありません。逆方向タイマーは、`0` で動作を停止します (負の値はありません)。

施工方法は2つあります。必要なものとセマンティクスが一致するものを選択してください。

- `create({ id, displayName?, direction?, currentMs?, scope?, domain? })` — **常に指定された初期値でタイマーを (再) 作成**し、`currentMs` を含む既存の状態を上書きします。 「新しく始める」という意味の場合にこれを使用します。ワンショットリセットブランチ内。
- `getOrCreateTimer({ id, displayName?, direction?, currentMs?, scope?, domain? })` — **冪等**。 `id` を持つタイマーがすでに存在する場合、その `displayName` および `direction` は更新される可能性がありますが、`currentMs` は保持されます。それ以外の場合は、指定された初期値を使用して作成されます。これは、一般的な「タイマーが存在することを確認してからタイマーを作動させる」パターンに必要なものです。

どちらのメソッドも、エンジンがルールの存続期間中記憶する 2 つの述語関数を受け入れます (これらはハートビート間および `webChangedEvent` 再評価間で存続しますが、ストレージに**永続化されることはありません**)。- `scope: (url) => boolean` — 各 `pageHeartbeatEvent` で現在表示されている URL が `true` の場合、タイマーはハートビート間隔 (~250 ミリ秒) で自動的に作動します。ヘルパー自体がブロックすることはありません。 `currentMs` のみを更新します。タイマーごとのハートビートごとに最大 1 つの自動ティック。
- `domain: (url) => boolean` — 現在表示されている URL が `true` の場合、タイマーはページ内オーバーレイ (左上) にレンダリングされます。 `domain` が省略された場合、エンジンは表示のために `scope` にフォールバックするため、追加の配線なしで「tick on /shorts/ Pages」タイマーもそこに表示されます。別の表示ゲートが必要な場合は、`domain` を明示的に指定します (たとえば、`/shorts/` のみにチェックマークを付けますが、`youtube.com` すべての残り時間を表示します)。

> **重要 — タイマーは単独でブロックすることはありません。** 逆方向タイマーがゼロに達すると、ゼロで停止し、`timerEnded` を 1 回起動します。実際にページをブロックするかどうかは、`helpers.getTimerHelper().isExpired(id)` をチェックした後に `ev.preventDefault()` を呼び出す別の `openWebEvent` / `switchWebEvent` ハンドラーによって決まります。この分離により、「警告のみ」のタイマー、カウントアップ トラッカー、ソフト ナッジ、ハード ブロックなど、同じプリミティブを自由に選択して構築できます。

その他の方法:

- `delete(id)`、`pause(id)`、`resume(id)` — 標準ライフサイクル。一時停止すると `currentMs` がフリーズします。
- `setDirection(id, "forward" | "backward")`、`setCurrentMs(id, ms)`、`addMs(id, deltaMs)` — 直接ミューテーター (ほとんどのルールではこれらは必要ありません。ハートビートでタイマーを刻ませます)。
- `setDisplayName(id, name)` — ラベルを変更します。
- `getCurrentMs(id)`、`getDirection(id)`、`getDisplayName(id)`、`isPaused(id)`、`exists(id)`。
- `isExpired(id)` — `currentMs === 0` の場合は `true`。
- `getState(id)` — `{ id, displayName, direction, isPaused, currentMs, isExpired }` または `null`。
- `list()` — このグループが状態オブジェクトの配列として所有するすべてのタイマー。

#### 11.3.2 `getPersistenceHelper()`

グループを範囲とするマップのようなストレージ。値は JSON シリアル化可能である必要があります。

- `set(key, value)`、`get(key, defaultValue?)`、`has(key)`、`delete(key)`、`keys()`、`entries()`、`clear()`、`size()`。

ソフト制限: グループあたり約 200 キー、値あたり 16 KB。

#### 11.3.3 `getLogHelper()`

- `log(...args)`、`warn(...args)`、`error(...args)` — ポップアップの **Log** パネルに書き込みます (ヘルパー バンドルは、どのディスパッチで生成されたかに関係なく、同じアキュムレーターを介してそれらをルーティングします)。各行には `[CustomBlocker:groupId]` というプレフィックスが付きます。
- ヘルパーにはハードキャップがあります。つまり、ディスパッチごとに約 **200 ログ エントリ**、エントリあたりの最大文字列長です。過剰なエントリは削除され、`accumulator.logsDropped` でカウントされます。これは、`for (let i = 0; i < 100000; i++) helpers.log(i)` の暴走からポップアップを保護するものです。
- **デバッグ モード** がオフ (デフォルト) の場合、エンジン自体が発行するトレース レベル エントリ (ディスパッチ開始 / ハンドラー タイミング) はどこでも抑制されます。ログ パネルには表示されず、コンソールにも出力されません。自分の `log` / `warn` / `error` 呼び出しは常に通過します。

#### 11.3.4 `getRedirectionHelper()`

現在のページがブロックされた場合にコンテンツ スクリプトが使用するリダイレクト URL を検査またはオーバーライドします。

- `get()` — このディスパッチの現在有効なリダイレクト URL を返します。最初は、これは組み込みグループの構成されたフォールバック URL (存在する場合)、そうでない場合は `""` です。
- `set(url)` — このディスパッチのリダイレクト URL をオーバーライドします。成功した場合は `true` を返し、文字列以外の入力の場合は `false` を返します。 `""` を渡すと、リダイレクト オーバーライドがクリアされ、通常のデフォルトの終了動作に戻ります。
- `createMessageUrl(message)` — `chrome-extension://<id>/message-page.html?msg=...` URL を返します。この URL に移動すると、メッセージがクリーンなページの中央に表示されます。タイマー終了後にユーザーを「出勤」/「休憩」画面にリダイレクトする場合に便利です。例: `ev.setRedirectLink(h.getRedirectionHelper().createMessageUrl("Go Work"))`。

他のカスタム ルールの副作用と同様に、この状態は現在のディスパッチのすべてのルールで共有されます。ルールは下から上に実行されるため、`set(...)` を呼び出す最上位のルールが優先されます。

#### 11.3.5 `getDomainHelper()` (エイリアス `getDomainUtility()`)

URL 検査ヘルパー。受信 URL はすでに newtab 正規化されているため、`normalize()` はありません。

コア：- `hostnameOf(url)`、`pathnameOf(url)`、`matches(hostname, site)`、`getPlatform(url)`。
- `isYouTubeHost`、`isTikTokHost`、`isInstagramHost`、`isFacebookHost`、`isTwitchHost`、`isRedditHost`、`isDiscordHost`。
- `youtube()`、`tiktok()`、`instagram()`、`facebook()`、`twitch()` — それぞれ `{ isPlatformUrl, isShortUrl, isVideoUrl, isPostUrl, isHomePage, extractAuthor, extractVideoId }` を返します。

URL フィルタリングとセクション ヘルパー:

- `isEmptyStartPage(url)` — 新しいタブ ページおよび同等のページの `true` (ハンドラーに対して `""` として表示される URL)。
- `matchesAny(url, patterns)` — `patterns` は、正規表現、文字列正規表現、またはそのいずれかの配列です。
- `pathStartsWith(url, path)` — 境界認識 (`pathStartsWith("/r/", "/r")` は true、`"/results/"` は非)。
- `queryHas(url, key, value?)`、`queryGet(url, key)` — クエリ文字列検査。
- `isSearchPage(url)` — Google / Bing / DuckDuckGo / ユーチューブ の結果 / レディット / ツイッター / X 検索を認識します。
- `isInfiniteFeedUrl(url)` — ユーチューブ、ティックトック、インスタグラム、フェイスブック、レディット、X のアルゴリズム フィード サーフェスを認識します。
- `sameSection(a, b)` — 同じホスト名であり、同じ最初のパス セグメント。

#### 11.3.6 `getPlatformHelper()`

プラットフォームごとの DOM インテントとサブセクション タイマー、および検査。各 `helpers.getPlatformHelper().<platform>()` は、メソッド セットが **プラットフォームによってゲートされている** オブジェクトを返します。特定のプラットフォームでは意味をなさないメソッドは単に存在しないため、それらを呼び出すと、サイレントに no-op するのではなく、`TypeError: ... is not a function` がスローされます。たとえば、`twitch().hidePosts` は存在しません (ツイッチ には投稿がありません)、`tiktok().hideShortButton` も存在しません (ティックトック のエクスペリエンス全体はすでに短編ビデオです)。実行時にイントロスペクトするには、`helpers.getPlatformHelper().hasMethod(platform, name)` または `.listMethods(platform)` を使用します。

> **述語の有効期間と単一スロット ルール。** `hideShorts` / `hideReels` / `hideClips` / `hideStreams` / `hideVideos` / `hidePosts` / `filterComments` / `filterLive` はそれぞれ、**1 つ**の永続述語を所有します。 `(group, platform, slot)`。述語のスコープは現在のイベントに限定されません。述語は、一度設定すると、一致する `show*()` が呼び出されるか、グループがアンロードされるまで、すべてのページのロードとすべてのディスパッチでアクティブのままになります。新しい関数を使用して同じメソッドを再度呼び出すと、前のメソッドが **置き換えられます**。エンジンは単一グループ内の複数の述語を OR マージしません。条件を結合するには、結合を自分で行う述語を 1 つ書きます。 `yt.hideVideos(item => isShort(item) || hasKeyword(item))`。 **異なる**グループ間で、各グループは独自の述語を提供し、いずれかのグループの述語が一致する場合、項目は非表示になります。

検査メソッドは、イベントにバンドルされているスナップショットからディスパッチ時に値を取得します。それらの可用性は、上記のマトリックスによって制御されます。

URL 分類子は、プラットフォームに関係なく常に再公開されます: `isPlatformUrl`、`isShortUrl`、`isVideoUrl`、`isPostUrl`、`isHomePage`、`extractAuthor`、`extractVideoId`。サブセクション タイマーは永続的なグループ バケットにタイマーを登録し、スコープが設定されている場合は、そのサブセクションに一致する URL のみをチェックします。タイマー メソッドは `{ id, direction, currentMs, displayName }` を受け入れ、同じプラットフォームごとのゲーティングに従います。

述語メソッドの場合、正規化された `item`: `{ url, name, author, length, views, publishedAt, description, live?, sponsored?, algorithmic? }` を使用して、一致するカードごとに述語が呼び出されます。どのフィールドも `null` にすることができます。 「有罪が証明されるまで無罪」 — 必要なフィールドが欠落している場合は、`false` を返します。

### 11.4 例

**簡単** — 平日の午前中に ユーチューブ ショート ページをブロックします。

```js
(event, helpers) => {
  const yt = helpers.getDomainHelper().youtube();

  function maybeBlock(ev) {
    if (!yt.isShortUrl(ev.url)) return;
    const { dayName, hour } = ev.time;
    const weekday = !["Saturday", "Sunday"].includes(dayName);
    if (weekday && hour >= 9 && hour < 12) {
      ev.preventDefault();
      ev.setResult(-1);
    }
  }

  event.registerOpenWebEvent("morning-block", maybeBlock);
  event.registerSwitchWebEvent("morning-block", maybeBlock);
}
```

**中** — ユーチューブ ショートの 1 日の予算 30 分。ショート URL が表示されている間、`pageHeartbeatEvent` でタイマーが自動的に作動します。タイマーがゼロに達すると、別のハンドラーがブロックを強制します。

## 13. 複数ページの動作- 同じグループ内で開いているすべてのタブは、同じタイマーを共有します。
- 同じグループ内のタブに切り替えると、そのオーバーレイがすぐに更新され、現在の共有時間が表示されます。
- カスタム ルール タイマーは、`pageHeartbeatEvent` によって駆動される **アクティブな表示** タブでのみ動作します。バックグラウンドのタブや最小化されたウィンドウでは、タブは進みません。これは、デフォルトのブロックグループのカウントダウンと一致します。
- 新しいルールが追加されると、開いているすべてのページが変更を検出し、数秒以内に再評価します。 **ただし**、新しく登録されたハンドラーは、すでに開いているタブを遡って「開く」ことはありません。そのため、実行するたびにポップアップにリロードのリマインダーが表示されます。
- ルールの有効期限が切れると、非表示のフィード カードとナビゲーション ボタンが次回の更新時に復元されます。

## 14. 設定

上部バーの歯車アイコンから [**設定**] ダイアログを開きます。

- **ハートビート間隔** — コンテンツ スクリプトがタブ時間を報告し、`pageHeartbeatEvent` を駆動する頻度。デフォルトは 250 ミリ秒です。値が低いほど応答性は高くなりますが、CPU の使用量が多くなります。
- **ティック間隔** — グローバル `tickEvent` が起動する頻度。デフォルトは 1000 ミリ秒です。
- **デバッグ モード** — デフォルトでは *オフ*。 *オン* の場合、エンジンはトレース レベルのエントリをログ パネル (`[trace] dispatchEvent`、`[trace] N handler(s)`) に出力し、`[CustomBlocker:trace]` 行をブラウザー コンソールに出力します。日常使用ではオフのままにしてください。不正な動作をするルールを診断するときにオンにします。 `pageHeartbeatEvent` は、デバッグ モードがオンの場合でもトレース ログから除外されます。これは、1 秒あたり 4 回起動され、残りがかき消されるためです。

## 17. プライバシーとストレージ

## 19. トラブルシューティング- **追加したグループは何も行いません。** グループが有効になっていること、スケジュールで許可されていること、アクティブなスヌーズがないこと、および (プラットフォーム グループの場合) ページが選択したコンテンツ タイプと作成者フィルターに実際に一致していることを確認してください。
- **1 つのタブでタイマーが停止しているか間違っています。** 切り替えて戻るか、タブにフォーカスします。これにより、共有タイマーからの強制更新がトリガーされます。
- **フィード カードを非表示にする必要があると考えた後、再び表示されます。** フィードの非表示は、ルールがアクティブにブロックしている間のみ実行されます。 `after-minutes` ルールがある場合、時間がゼロになるとフィードの非表示が開始されます。
- **非表示になるはずだった ユーチューブ ナビゲーション ボタンがまだ存在しています。** ナビゲーションを非表示にするには、ルールを「作成者でフィルタしない」に設定し、コンテンツ タイプをショートまたは ユーチューブ 投稿にする必要があります。作成者フィルターを使用すると、非表示はカードごとにのみ行われます。
- **カスタム ルールは何もしないか、サイレントにスローされました。** 設定を開き、**デバッグ モード**を有効にし、もう一度**実行**をクリックして、ログ パネルを確認します。 `[trace]` というプレフィックスが付いた行には、すべてのディスパッチとハンドラーが表示されます。 `helpers.getLogHelper().log(...)` を使用して、独自のトレース ポイントを追加します。不正な動作をするルールが自動隔離され続ける場合は、ソースを修正して [実行] をクリックします。実行すると中止の理由がクリアされます。
- **新しいカスタム ルールは、すでに開いているタブには影響しません。** タブをリロードします。カスタム ルールは *将来の* ページ イベントに付加されます。ポップアップには、実行するたびにリロードするようリマインダーが表示されます。
- **カウントダウン タイマーが進みません。** カスタム ルール タイマーは、`pageHeartbeatEvent` を介して **アクティブな表示** タブでのみ動作します。バックグラウンド タブ、最小化されたウィンドウ、ロックされた画面は仕様により一時停止します。これは、デフォルトのブロック グループのカウントダウンと同じ動作です。
- **グループを削除できません。** 凍結されている可能性があります。厳密に凍結されたグループは、ロックが期限切れになるまではまったく削除できません。非厳密に凍結されたグループは、凍結解除の儀式を通じて削除できます。
- **ポップアップに「実行中…」と永久に表示されます。** カスタム ルールがタイトなループに入った可能性があります。エンジンはハード タイムアウト後にルールを強制終了し、ルールを隔離します。中止理由を確認するには、「ログ」パネルを開きます。ルールを修正し、「実行」をクリックします。

## 20. 用語集

- **ブロック グループ** — 独自のタイプ、動作、スケジュール、フリーズ/スヌーズを備えた 1 つのルール セット。
- **即時ブロック** — ルールはアクティブになるとすぐにブロックされます。
- **分後のブロック** — ルールは、その期間の時間予算が使い果たされた後にのみブロックを開始します。
- **リセット間隔** — 分後の予算をリセットする頻度。
- **スケジュール** — グループがアクティブになる日数と時間枠。
- **凍結 / 厳密な凍結** — 改ざん防止状態。
- **スヌーズ** — 構成可能な確認儀式により一時的に無効になります。
- **作成者フィルター** — プラットフォーム グループの場合、ルールを特定のコンテンツ作成者に制限します。
- **コンテンツ タイプ** - プラットフォーム グループの場合、ルールを特定の形式のコンテンツ (短い、長い、投稿) に制限します。
- **ヘルパー** — カスタム ルールのハンドラーに渡されるユーティリティ。
- **プラットフォーム** — `youtube`、`tiktok`、`facebook`、`instagram`、`twitch`のいずれか。それぞれに独自のグループ タイプとフィード非表示ロジックがあります。
- **ハートビート** — アクティブな表示タブから送出される約 250 ミリ秒の `pageHeartbeatEvent`。
- **チェックマーク** — 1 s グローバルに共有される `tickEvent` (可視性非依存)。
- **デバッグ モード** — 内部トレースのログをログ パネルとブラウザ コンソールに表示する設定。
- **隔離** — 実行時の安全制限 (期限、スパムのログなど) を超えたカスタム ルールを自動的に無効にします。次の実行でクリアされます。

## 21. 制限事項- フィードの非表示は、各プラットフォームの現在の DOM によって異なります。プラットフォームのレイアウトが変更された場合は、非表示セレクターを更新する必要がある場合があります。
- ユーチューブ 以外のサイトのプラットフォーム コンテキスト検出はほとんどが URL ベースであるため、正規のコンテンツ URL に対して最も信頼性が高くなります。
- カスタム ルール タイマーはハートビート解像度 (~250 ミリ秒) で動作します。 1 秒未満のタイミングについてはそれらに依存しないでください。
- `hideShorts` / `hideVideos` / `hidePosts` に渡された述語は、フィード カードごとに同期的に評価されます。述語内の重いロジックにより、フィードのスクロールが遅くなる可能性があります。安くしておいてください。
- 同じグループごとのタイマーを編集する 2 つのタブは、「最後の書き込み優先」戦略を同時に使用します。通常の使用にはこれで問題ありません。正確な計算に依存している場合は、時折小さなずれが発生することが予想されます。
- ブラウザーは、アイドル時にバックグラウンド Service Worker を一時停止する場合があります。拡張機能は、ページまたはアラームで必要になるとすぐに再開します。サイト/時間指定の使用量予算は、ハートビート リプレイを通じてカウントされ続けます。

## v1.2 メモ

カスタムルールエディターは スクリプト言語 構文を色分けするようになり、テンプレートブラウザーのコードプレビューも同じ色を使います。グループの一括操作は **クリア** と呼ばれます。
